# Go-to-Forth Transpiler: Architecture & Pipeline Plan

## 1. The Architecture (go2json + JS Emitter)
To rapidly bootstrap the transpiler, we will use **[go2json](https://github.com/CreativeInquiry/go2json)** to handle the heavy lifting of parsing Go code. `go2json` takes Go source code and outputs a standard JSON representation of the Go AST.

### The Pipeline Flow
1.  **The UI:** The user writes Go code in the React app.
2.  **The Push:** The React app pushes the `main.go` file, a pipeline YAML, and a `headless-test.js` script to a temporary GitHub branch.
3.  **The Pipeline (GitHub Actions):**
    *   Downloads `go2json.js` (a zero-dependency vanilla JS script).
    *   Runs `node go2json.js main.go ast.json`.
    *   Runs `node headless-test.js`.
4.  **The Emitter (JS):** 
    *   `headless-test.js` reads `ast.json`.
    *   It walks the JSON AST and emits Forth code.
    *   It executes the Forth code (eventually in a headless browser/worker environment).
    *   It captures the output as an artifact.

### Why this approach?
*   **Zero Parsing Logic:** We don't have to write a complex Go lexer or parser. We rely on `go2json` which handles the tokenization and AST generation natively in JavaScript.
*   **Zero Go Toolchain:** Because `go2json` is written entirely in JS, we don't even need to install Go in our pipeline. The entire transpilation process from Go to Forth happens in Node.js!
*   **Fast Iteration:** We only need to write the AST-to-Forth emitter in JavaScript, which is easy to debug and iterate on.

## 2. Concurrency Model: Instance-per-Goroutine
To support Go's concurrency (`go func()`) and channels, we will use a **WebAssembly Actor Model**. Instead of writing a complex scheduler in Forth, we map Goroutines directly to WebAssembly Instances.

### The WASM Footprint
*   **The Module:** A minimal Forth kernel compiled to WASM is ~15KB. We compile this `WebAssembly.Module` exactly once.
*   **The Instance:** Every time the `go` keyword is invoked, we create a new `WebAssembly.Instance` from the module. Instantiation takes microseconds.
*   **The Cost:** Each instance requires a minimum of 1 WASM Memory Page (64KB). 1,000 concurrent goroutines will consume ~64MB of RAM, which is highly efficient for a browser.

### The Split Memory Architecture
To support both isolation and communication, we use a split memory model:

1.  **Personal Memory (Private):** 
    *   Each Forth instance (Goroutine) has its own isolated 64KB memory page.
    *   This holds the Data Stack, Return Stack, and local variables.
    *   Goroutines cannot corrupt each other's execution state.
2.  **Shared Memory (The Heap):**
    *   We allocate a single `WebAssembly.Memory` object (backed by a `SharedArrayBuffer`) in the JavaScript host.
    *   This shared memory is imported into *every* Forth instance.
    *   **Usage:** Global variables, large structs, and **Channels** live here.
    *   **Synchronization:** Forth instances use WASM atomic operations (`i32.atomic.wait` and `i32.atomic.notify`) to safely read/write to channels in the shared memory.

## 3. Future Work: Self-Hosting
Once the AST-to-Forth emitter is robust and supports a sufficient subset of Go ("Go-Lite"), we can rewrite the emitter itself in Go. 

At that point, we can feed the Go-based emitter into itself to produce a Forth-based transpiler, achieving full self-hosting. For now, the JS emitter + `go2json` is the pragmatic stepping stone.

## 5. The AST-to-Forth Emitter
The Emitter is the core of our transpiler. It consumes the JSON AST generated by `go2json` and produces Forth code. To maintain high performance, it enforces a strict architectural split between internal concurrency and system I/O.

### The Architectural Split: "Fast Path" vs. "Slow Path"

#### 1. The "Fast Path" (Internal Concurrency)
*   **What it is:** Channel operations (`ch <- x`, `<-ch`), shared memory access, and goroutine scheduling.
*   **Where it lives:** Entirely within the browser's memory space, managed by our transpiler's runtime.
*   **How it works:** We use `SharedArrayBuffer` and `Atomics` to implement channels. When a goroutine sends data on a channel, it performs an atomic write to the shared memory and calls `Atomics.notify`.
*   **Performance:** Nanoseconds. No kernel, no VFS, no syscalls.

#### 2. The "Slow Path" (Wanix I/O)
*   **What it is:** `fmt.Println`, reading files, network requests.
*   **Where it lives:** The Wanix kernel.
*   **How it works:** When a goroutine needs to perform I/O, it makes a standard WASI syscall. Wanix intercepts this, handles the VFS/security check, and performs the actual browser-native I/O operation.
*   **Performance:** Milliseconds (acceptable for I/O).

### Emitter Logic
The Emitter walks the JSON AST recursively.

*   **For `CallExpr` (Concurrency):** If the expression is a channel operation, the Emitter generates Forth macros that map directly to `SharedArrayBuffer` offsets and `Atomics` instructions.
*   **For `CallExpr` (I/O):** If the expression is an I/O call (e.g., `fmt.Println`), the Emitter generates Forth macros that call a `HOST-LOG` word. This word triggers the standard WASI/Wanix syscall for `stdout`.

### Why this split?
1.  **Separation of Concerns:** Wanix handles the "OS" (isolation, security, I/O). Our Forth/WASM runtime handles the "Language" (concurrency, memory management, scheduling).
2.  **Performance:** We avoid the overhead of the kernel for the most frequent operations (channel communication).
3.  **Portability:** If we ever decide to move away from Wanix, our internal concurrency model remains exactly the same. We only have to swap out the I/O syscall implementation.
